# PRD: Secure LAN Pi Observer Subscriptions

## Problem

The Secure LAN Pi communication system can send prompts between trusted hubs,
but agents also need a way to wait for work completion, state changes, or other
signals without polling. A polling model wastes tokens, CPU, network traffic, and
operator attention. Direct agent-to-agent watching would also weaken the hub
security model by creating many small network endpoints and authorization paths.

The goal is to add hub-mediated observer subscriptions: an authenticated,
WebSocket-delivered publish/subscribe layer where one agent can subscribe to
specific events from another agent, hub, task, or project and be notified when a
matching event occurs.

## Users / Jobs To Be Done

- Primary user: A Pi operator coordinating multiple agents across one or more
  trusted LAN machines.
- Job/story: As an agent, I want to subscribe to meaningful events, such as
  another agent completing a task, so I can sleep while idle and resume only when
  useful work is available.
- Job/story: As an operator, I want event delivery to preserve the Secure LAN Pi
  trust model, so untrusted hubs cannot observe or trigger private agent
  workflows.
- Current workaround: Agents poll task state, wait for direct prompt responses,
  or rely on a human to manually resume dependent work.

## Goals

1. Add a hub-mediated observer subscription system to `coms-lan.ts`.
2. Allow local agents to subscribe to event patterns through their local hub.
3. Allow trusted hubs to advertise and route subscriptions across authenticated
   `wss://` connections.
4. Deliver matching events to subscribed agents without polling.
5. Support durable delivery for short disconnects using event IDs, acknowledgments,
   leases, and replay cursors.
6. Preserve the security model from the parent Secure LAN Pi communication PRD.
7. Provide audit logs for subscription lifecycle, event publication, routing,
   delivery, acknowledgments, retries, expiry, and authorization failures.

## Non-Goals

- No Internet-scale distributed queue in v1.
- No anonymous peer-to-peer routing.
- No DHT, Chord, Pastry, Kademlia, or global rendezvous overlay in v1.
- No full RabbitMQ, AMQP, Kafka, NATS, or MQTT compatibility layer in v1.
- No direct agent-to-agent sockets.
- No delivery to untrusted hubs or agents.
- No automatic restart of stopped agent processes in v1.
- No subscription matching on raw prompt bodies by default.
- No guaranteed exactly-once delivery in v1.
- No multi-hop routing unless the parent PRD later adds trusted multi-hop hub
  routing.

## Requirements

### Functional Requirements

- Implement observer subscriptions as an extension of the authenticated
  `coms-lan.ts` hub protocol.
- Agents must create subscriptions through their local machine hub.
- Hubs must deliver subscription events over existing local agent delivery
  channels when the subscriber is connected.
- Hubs must queue matching events for offline-but-registered subscribers until
  the subscription lease, event retention window, or queue limit expires.
- Hubs must not attempt to restart a stopped agent process in v1.
- Subscriptions must include a stable subscription ID.
- Subscriptions must include one or more event types.
- Subscriptions may include structured filters such as producer agent ID, hub ID,
  session ID, project label, task ID, correlation ID, and event metadata.
- Subscriptions must include a lease duration.
- Hubs must expire subscriptions whose leases are not renewed.
- Agents must be able to unsubscribe explicitly.
- Agents must be able to renew a subscription before lease expiry.
- Events must include a stable event ID.
- Events must include an event type.
- Events must include producer identity metadata.
- Events must include a timestamp generated by the publishing hub or local agent.
- Events must include an optional correlation ID for request, task, or workflow
  linkage.
- Events must distinguish control metadata from payload data.
- Hubs must support at-least-once delivery for subscribed events.
- Subscribers must acknowledge delivered events.
- Hubs must retry unacknowledged deliveries until retry limits, lease expiry, or
  retention expiry.
- Subscribers must be able to reconnect with a `resumeFromEventId` cursor for a
  known subscription.
- Hubs must de-duplicate event publication and delivery by event ID.
- Hubs must support local-only subscriptions for events produced by local agents.
- Hubs must support trusted remote subscriptions for events produced by agents on
  an authenticated remote hub.
- Hubs must advertise subscription interest only to trusted hubs that are allowed
  to observe the requested event scope.
- Hubs must not send all local events to every trusted hub by default.
- Remote subscription advertisements must avoid leaking raw prompt text, secrets,
  private local paths, or unnecessary working-directory details.
- Hubs must authorize subscription creation before accepting it.
- Hubs must reject subscriptions from untrusted hubs.
- Hubs must reject subscriptions whose filters request unauthorized event scopes.
- Hubs must reject malformed event types, invalid filters, expired leases, and
  unknown protocol versions.
- Hubs must audit subscription creation, renewal, unsubscribe, expiry, denial,
  event publication, event routing, event delivery, acknowledgement, retry, and
  drop decisions.

### Event Types

V1 should reserve a small set of event types and allow future namespaced event
families.

Required v1 event types:

- `agent.task.started`
- `agent.task.completed`
- `agent.task.failed`
- `agent.signal`
- `agent.response.ready`
- `hub.peer.trusted`
- `hub.peer.disconnected`

Event names must use lowercase dotted namespaces.

### Example Subscription Message

```json
{
  "type": "subscribe",
  "protocolVersion": "coms-lan/1",
  "subscriptionId": "sub_01J...",
  "eventTypes": ["agent.task.completed", "agent.task.failed"],
  "filters": {
    "producerAgentId": "agent-b",
    "correlationId": "task_01J..."
  },
  "leaseSeconds": 3600,
  "resumeFromEventId": "evt_01J..."
}
```

### Example Event Message

```json
{
  "type": "event",
  "protocolVersion": "coms-lan/1",
  "eventId": "evt_01J...",
  "eventType": "agent.task.completed",
  "producerHubId": "hub_01J...",
  "producerAgentId": "agent-b",
  "correlationId": "task_01J...",
  "timestamp": "2026-05-22T12:00:00.000Z",
  "payload": {
    "status": "success",
    "summary": "Task completed"
  }
}
```

### Example Delivery Acknowledgement

```json
{
  "type": "ack",
  "protocolVersion": "coms-lan/1",
  "subscriptionId": "sub_01J...",
  "eventId": "evt_01J..."
}
```

## Non-Functional Requirements

- Secure by default: only trusted authenticated hubs may exchange subscription
  advertisements or event deliveries.
- Minimal dependency footprint.
- Deterministic matching behavior for event filters.
- Bounded memory and disk usage for subscription state and event retention.
- Idempotent event handling using event IDs and delivery IDs where needed.
- Cross-platform persistence under `~/.pi/coms-lan/`.
- Clear operator-facing diagnostics for denied subscriptions and dropped events.
- No sensitive payloads in UDP discovery packets.
- No raw prompt bodies in audit logs by default.
- No hidden fallback that silently downgrades delivery guarantees.

## References / Prior Art

### Parent Secure LAN Pi Communication PRD

- `PRD.md` in this directory defines hub discovery, authentication, trust, and
  direct prompt send/await behavior. This feature must build on that security
  model instead of creating separate agent network endpoints.

### Academic / Conceptual Prior Art

Use these as design references during planning. The v1 implementation should
borrow concepts, not implement a full academic-scale overlay.

- SIENA: scalable Internet event notification architectures and content-based
  publish/subscribe routing.
- Gryphon: content-based publish/subscribe over broker networks.
- Hermes: distributed event-based middleware with event routing.
- Scribe/Pastry: topic-based decentralized publish/subscribe over a structured
  peer-to-peer overlay.
- Bayeux/Tapestry: application-level multicast over structured overlays.
- Gossip and epidemic pub/sub systems: useful for understanding dissemination
  tradeoffs, but not the preferred v1 event delivery model.
- SWIM/memberlist: useful for membership and failure detection, but not a
  substitute for subscription routing.

### Practical Messaging Prior Art

- RabbitMQ / AMQP: brokered queues, acknowledgments, routing keys, and durable
  delivery concepts.
- NATS: lightweight pub/sub and request/reply concepts.
- MQTT: topic subscriptions, retained messages, and session behavior.
- Kafka: durable event log and replay cursor concepts.

## Recommended Design Direction

Use a simplified broker overlay:

```text
Agent A -> Local Hub A: subscribe to agent.task.completed
Hub A -> Trusted Hub B: advertise subscription interest if producer is remote
Agent B -> Local Hub B: publish agent.task.completed
Hub B -> Hub A: route matching event over authenticated WSS
Hub A -> Agent A: deliver event
Agent A -> Hub A: ack event
Hub A -> Hub B: ack routed event when appropriate
```

Prefer direct trusted-hub routing in v1. Do not introduce DHT-based routing or
multi-hop forwarding unless later requirements prove direct routing insufficient.

## Acceptance Criteria

1. [ ] A local agent can subscribe to a local event without polling.
   - Verify: Start two local agents registered with one hub. Agent A subscribes
     to Agent B's `agent.task.completed` event. Agent B publishes the event.
   - Pass: Agent A receives the event over its local delivery channel and ACKs it.
   - Fail: Agent A must poll, misses the event, or receives duplicate deliveries
     after ACK.

2. [ ] A trusted remote hub can route a matching event to a subscriber.
   - Verify: Start two authenticated trusted hubs. Agent A on Hub A subscribes to
     an event from Agent B on Hub B. Agent B publishes a matching event.
   - Pass: Hub B routes the event to Hub A only after authentication and Hub A
     delivers it to Agent A.
   - Fail: The event bypasses hub authentication, is broadcast to unrelated hubs,
     or is not delivered.

3. [ ] Untrusted hubs cannot create observer subscriptions.
   - Verify: Attempt to subscribe from an untrusted discovered hub.
   - Pass: The subscription is rejected and audited without exposing event data.
   - Fail: The untrusted hub receives events, subscription metadata, or agent
     listings.

4. [ ] Subscription leases expire abandoned observers.
   - Verify: Create a short-lived subscription and stop renewing it.
   - Pass: The hub expires the subscription, stops matching new events, and logs
     the expiry.
   - Fail: The subscription remains active indefinitely.

5. [ ] At-least-once delivery with ACK and retry works for transient disconnects.
   - Verify: Deliver an event while the subscriber disconnects before ACK, then
     reconnect with the same subscription and replay cursor.
   - Pass: The hub retries or replays the event and stops retrying after ACK.
   - Fail: The event is permanently lost before retention expiry or retries
     forever after ACK.

6. [ ] Event retention is bounded.
   - Verify: Configure a small retention limit, publish more events than the
     limit, and inspect retained state.
   - Pass: Old events are dropped according to documented retention policy and
     drops are audited when relevant.
   - Fail: State grows without bound or drops are silent.

7. [ ] Audit logs cover the observer lifecycle.
   - Verify: Trigger subscribe, renew, publish, route, deliver, ACK, retry,
     unsubscribe, expiry, denied subscription, and dropped event paths.
   - Pass: Each event type appears in audit logs with useful non-secret metadata.
   - Fail: Required events are missing or logs include secrets/private key
     material.

8. [ ] Event filters do not match unauthorized scopes.
   - Verify: Attempt to subscribe to a producer, project, or event type outside
     the caller's allowed scope.
   - Pass: The hub rejects the subscription and audits the denial.
   - Fail: The caller receives events outside its authorized scope.

## Alternatives Considered

| Option | Pros | Cons | Decision |
|--------|------|------|----------|
| Local polling | Simple to implement | Wastes resources and delays dependent work | Rejected |
| Direct agent-to-agent watches | Avoids hub broker logic | Many endpoints, weak auth story, hard auditing | Rejected |
| Central RabbitMQ/NATS/MQTT broker | Mature queue behavior | Adds external dependency and central service | Rejected for v1 |
| Full DHT-based pub/sub mesh | Scales beyond LAN | Too complex for trusted LAN MVP | Deferred |
| Gossip-based event fanout | Simple dissemination | Over-delivers events and complicates privacy | Rejected for v1 delivery |
| Hub-mediated broker overlay | Matches existing hub model, auditable, secure | Requires subscription state and retry logic | Accepted |
| Exactly-once delivery | Strong user-facing semantics | Expensive and hard in distributed systems | Rejected for v1; use at-least-once |

## Risks

| Risk | Impact | Mitigation |
|------|--------|------------|
| Subscription filters leak sensitive workflow details | Remote hubs learn private intent | Authorize scopes, minimize advertisements, avoid prompt bodies |
| Event payloads leak secrets | Sensitive data reaches subscribers or logs | Separate metadata from payload, redact audits, default to summaries |
| Duplicate deliveries cause repeated agent actions | Agents repeat work or spend tokens twice | Require event IDs, ACKs, and idempotent subscriber behavior |
| Unbounded queues grow on inactive subscribers | Disk/memory exhaustion | Use leases, retention windows, queue limits, and audited drops |
| Retry storms after network failure | LAN and agent overload | Backoff retries and cap pending deliveries |
| Authorization model becomes too coarse | Trusted hub can observe too much | Add event-scope authorization before remote subscriptions |
| System becomes a full message broker project | Scope creep delays Secure LAN Pi | Keep v1 direct, authenticated, bounded, and LAN-focused |

## Open Questions

- What default lease duration should observer subscriptions use?
- What default event retention window and queue size should hubs use?
- Should remote subscription authorization be per trusted key, per hub ID, per
  project label, per event namespace, or a combination?
- Should prompt response readiness use `agent.response.ready`, direct request
  correlation, or both?
- Should event payloads be encrypted end-to-end for the subscribing agent, or is
  hub-to-hub WSS plus hub trust sufficient for v1?
- Should subscription filters support exact-match only in v1, or include simple
  wildcard event namespaces such as `agent.task.*`?
- Should remote hubs receive negative acknowledgments for permanently dropped
  deliveries?
